# ADR-001: JSON-compatible YAML 作为状态文件格式

> 状态: 已接受
> 日期: 2026-04-22

## 1. 背景

项目需要一个可人工阅读、可机器解析的状态文件格式来存储小说创作的结构化数据（实体、大纲、投影等）。

## 2. 决策

使用 `.yaml` 后缀但内容必须为合法 JSON。

## 3. 决策原因

- JSON 是最通用的数据交换格式，Python stdlib 原生支持
- `.yaml` 后缀暗示人类可读性，但避免引入 YAML 解析器依赖
- 实际用 `json.load` / `json.dump` 处理，零外部依赖
- 保持与 AI agent 生态的兼容性（大多数 agent 原生理解 JSON）

## 4. 影响

### 正面影响

- 零外部依赖，符合项目的 stdlib-only 约束
- 文件可人工阅读和编辑
- 机器解析稳定可靠

### 负面影响

- 不支持 YAML 的锚点、引用等高级特性
- 不支持多行字符串的优雅写法
- 需要开发时注意不要意外使用 YAML 特有语法

## 5. 被放弃的备选方案

### 备选方案 A: 标准 YAML

- 为什么没选: 需要 `PyYAML` 依赖，违反零依赖约束

### 备选方案 B: JSON 文件

- 为什么没选: `.json` 后缀在 AI agent 生态中暗示 API 响应而非配置，且缺少人类友好的多行支持

### 备选方案 C: TOML

- 为什么没选: Python 3.10 的 `tomllib` 只支持读不支持写

## 6. 迁移或落地要求

- 所有 `protocol/io.py` 中的读写函数已按此规则实现
- `docs/ARCHITECTURE_GUARDRAILS.md` 已标注此约束为禁止事项

## 7. 何时需要重新审查

- 项目决定引入外部依赖时
- 需要 YAML 高级特性（锚点、多文档等）时
- Python 版本要求变化影响 stdlib JSON 能力时
