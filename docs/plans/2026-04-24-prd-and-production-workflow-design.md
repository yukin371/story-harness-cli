# PRD 模板与规格化产出流程设计

> 日期: 2026-04-24
> 状态: 设计完成，待实施
> 关联: docs/templates/PRD.template.md

## 1. 背景

Story Harness CLI v0.2 已完成全部 P0-P3 路线，工具能力覆盖了从初始化到导出的完整闭环。但在实际使用中暴露了一个核心问题：

**项目产出物组织混乱。**

用户把"小说 + AI 生成工具"看作一个整体项目，但目前仓库里：
- CLI 工具代码、小说产出物、文档混在一起
- 小说工程内部缺少"产品需求"视角的统合文档
- 规格文件（大纲、人物）在长篇场景下会膨胀到 AI 上下文窗口放不下
- 没有标准化的产出流程和验收条件

## 2. 核心设计决策

### 2.1 小说工程 = 软件项目

建立一套类比映射，让小说创作的管理方式与软件工程对齐：

| 软件项目 | 小说项目 |
|---------|---------|
| PRD | 小说产品定义（题材、平台、受众、卖点） |
| 架构设计 | 大纲、世界观、人物设定 |
| 源代码 | 正文（chapters） |
| 单元测试 | 章节评审（review chapter/scene） |
| CI/CD | doctor 检查、质量门禁 |
| 发布包 | 导出产物（manuscript） |
| 运行日志 | 工作流日志 |

### 2.2 PRD 以小说产品为中心

每个小说工程有自己的 PRD.md，定义：
- 产品是什么（题材、平台、受众）
- 商业连载蓝图（连载模型、字数目标、钩子策略）
- 产出物标准（文件组织、质量门槛）
- 验收标准（单章/单卷/完结的交付条件）

### 2.3 短篇/长篇双模式文件组织

- **短篇**（< 5 万字）：扁平结构，单文件规格足够
- **长篇**（>= 5 万字）：按卷拆分，索引 + 分卷详情
  - 大纲：`outline.yaml` 索引 + `outlines/vol-*.yaml`
  - 人物：`entities.yaml` 索引 + `entities/*.yaml`
  - 时间线：`timeline.yaml` 索引 + `timelines/vol-*.yaml`

拆分触发条件：单文件超过 ~500 行或 ~20KB。

### 2.4 AI 上下文窗口适配

拆分的直接收益是 AI 上下文管理：
- `context refresh` 只加载当前卷的大纲切片
- 只加载当前活跃角色的详细设定
- 按需加载时间线事件，而不是全量灌入

## 4. v0.3 影响分析

本设计对 v0.3 有以下影响：

### 4.1 协议层变更（P0 — 必须做）

- `protocol/state.py` 需要支持按卷拆分的文件路径
- `init` 命令需要支持 `--prd` 标志生成 PRD.md
- 所有命令的 `--root` 路径解析需要适配 spec/drafts/workflow/exports 分层

### 4.2 context refresh 增强（P1 — 应该做）

- 按卷加载大纲，而不是全量
- 按活跃角色加载设定，而不是全量
- 智能选择加载的上下文窗口大小

### 4.3 迁移支持（P1 — 应该做）

- 提供 `migrate` 命令将旧扁平结构迁移到新分层结构
- 保持向后兼容：旧结构仍然可用，新结构可选启用

### 4.4 模板验证（P2 — 可以做）

- 基于 demo-urban-occult-long 创建试点 PRD
- 验证模板的实用性和完整性
- 收集反馈后迭代模板

## 5. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 文件路径变更破坏现有项目 | 高 | 新结构可选启用，旧结构继续支持 |
| 拆分粒度不当导致频繁跨文件查询 | 中 | 索引文件提供快速查找，context refresh 做智能聚合 |
| PRD 模板过于复杂导致不实用 | 低 | 先在试点项目验证，简化后再推广 |

## 6. 不做的事

- 不改变 project.yaml 的现有字段结构（PRD.md 与 project.yaml 互补，不替代）
- 不引入数据库或非文件存储
- 不做 UI/编辑器
